# z-spanish 调整补充（基于词块记忆讨论）

本文档是对原始 z-spanish Core Basics 设计的**补充调整**，不改变整体架构，只针对本次讨论中发现的问题做修正。

---

## 一、词块颗粒度：三层结构必须整合，不能停在最小粒度

基础库不能大部分内容都是"冠词+名词"这一种小块，必须确保三层都有覆盖：

1. **最小词块**：冠词+名词（el agua / la mesa）
2. **动词固定搭配**：tener que / ir a / hace falta 等介词搭配
3. **完整功能句**：No hace falta, gracias / ¿Puedo pagar con tarjeta?

原文档第 10、19 节已经在做人称动词块和场景句，方向对，但要在内容规划阶段明确检查三层比例，避免最终库里"冠词+名词"占比过高、动词搭配和整句偏少。

---

## 二、颜色区分阴阳性：撤回，不做

原计划"基础库用颜色区分阴阳性、训练页不用"，重新考虑后撤回。

**原因**：冠词+名词已经用融合词块样式呈现（el/la 与名词粘成一个不可拆分的整体），这个设计本身就是让人不再需要单独判断"这是阳性还是阴性"——词块作为一个整体被记住,性别信息已经内嵌在词块形态里。额外加颜色区分，等于又把"性别"从词块里单独拎出来强调，和词块记忆"不拆分、整体记"的初衷相矛盾，属于不必要的重复设计。

**最终方案**：冠词词块统一用一种中性样式呈现，不做阳性/阴性的颜色区分。

---

## 三、特例词单独标记

`el agua` 类词（阴性名词但因发音规则单数用 el：el agua、el águila、el hacha）需要：

- 单独的 `confusion_group` 标签，例如 `"el_femenino_fonetico"`
- note 中说明原因（重读 a 开头 + 阴性名词，复数/形容词仍按阴性：las aguas, el agua fría）

不要和真正的阳性名词混在同一分组里。

---

## 四、判定宽松化（near_correct）

现有 correct / near_correct / wrong / forgot 四档基本不变，但需要放宽 near_correct 的判定范围：

- 不区分大小写
- 无重音符号（á→a 等）也计入 near_correct，而不是 wrong

原因：技术性错误判太严会造成"明明会但系统说我错"的挫败感，对长期使用意愿的伤害大于宽松判定带来的准确性损失。

---

## 五、统计页：去掉"连续学习天数"

原文档统计页里的"连续学习：7 天"改为：

```
本月完成天数：18 天
```

或"近30天完成次数"，与 ZBody 项目已经确认的设计原则保持一致——避免 streak 机制导致的全或无放弃模式。

---

## 六、"需要加强"设软上限

原文档"如果旧内容超过 20：今天不加新内容"这条规则之外，再加一条：

- **需要加强池超过 15 条时，当日训练优先清空该池，不再新增"需要加强"内容**（除非当天本来就答错触发新的加入）

避免这个池子无限增长成为焦虑源。

---

## 七、反馈页 note 长度克制

答题反馈页（正确/需要加强）里的 note 说明保持 1 句话以内，不做语法展开讲解。一旦开始展开会打断训练节奏、引发分心退出。

---

## 八、（建议评估）每日固定时间提醒

这条不是新增复杂功能，而是评估要不要加一条最简单的系统级推送提醒（固定时间，无需用户设置），因为"记得打开app"本身对 ADHD 是比"打开后能不能坚持练"更大的门槛。如果暂时不做，也应作为已知取舍记录在案，而不是遗漏。

---

## 九、熟练度系统（复用间隔复习节奏，不新增评分逻辑）

**背景**：需要一个客观依据来判断"这个词块到底会不会"，不依赖自己的主观感觉。

**设计原则**：不做百分比/评分/进度条，避免制造新的焦虑源。直接复用已有的间隔复习节奏，把复习进度转成阶段标签：

| 间隔阶段 | 熟练度标签 |
|---|---|
| 今天 / 第2天 | 初识 |
| 第4天 / 第7天 | 巩固中 |
| 第14天 / 第30天 | 稳固 |
| 第60天 / 第90天 | 长期记忆 |

**规则**：
- 阶段判定完全基于硬数据：到达该间隔时答对（correct 或 near_correct）才算通过该阶段
- 答错或"我不会"→ 阶段**回退一级**（不清零），如实反映记忆状态，不做惩罚性文案
- 每条词块的熟练度阶段独立计算，互不影响

**显示位置**：
- **基础库列表**：显示阶段标签（如"稳固"），作为快速浏览的参考
- **训练页**：不显示。训练时提前看到"这题应该会"会干扰主动回忆的诚实性，违背主动输出优先原则
- **统计页**：可选加一行"当前长期记忆词块数"作为整体进度参考，仍不使用百分比

## 十、高频词表的正确用法：只做抽取标准，不做内容形式

高频词表（动词/名词/介词）本身是孤立单词清单，**不能直接搬进基础库**，只能用来检查"哪些高频项还没被词块覆盖"，查出缺口后按下列规则补，补的是词块，不是单词。

### 10.1 高频动词

对照 A1-A2 高频动词表，每个动词落地时必须满足：

- 至少 1 条**人称变位块**（quiero / quieres / quiere / queremos，对应原文档第10节）
- 至少 1 条**动词+介词/搭配块**（tener que、ir a 等，对应原文档第15节）
- 至少 1 条生活场景例句（可以复用 example_es 字段，不必单独开 entry）

同一动词若只有孤立的"quiero = 我想要"这一条，视为**未完成收录**，需要补齐搭配块和例句。

### 10.2 高频名词

对照高频名词表，每个名词落地时必须满足：

- 1 条**冠词词块**（el agua / la mesa，对应原文档第12节）
- 至少 1 条**动词+名词搭配句**（Necesito agua. / Voy a la tienda.）
- 若属于特例词（el agua 类），归入对应 confusion_group

同一名词若只有孤立的"el/la + 名词"这一条，没有搭配句，视为未完成收录。

### 10.3 高频介词

介词永远不单独出现，只以短语/搭配形式收录（延续原文档第8节原则），落地时必须满足：

- 至少 2 条不同搭配短语（例如 para：para llevar / para mí）
- 若属于易混介词对（por/para、a/en、de/desde），必须归入 confusion_group，并在 note 里用一句话点出区分依据，不展开语法讲解

### 10.4 校验规则（写入 JSON 收录检查）

- `example_es` 字段永远不能为空——已有规则，天然阻止孤立收录
- 动词类词条：同一动词至少 2 条 entry（人称块 + 搭配/例句），否则标记未完成
- 名词类词条：同一名词至少 2 条 entry（冠词块 + 搭配句），否则标记未完成
- 介词类词条：同一介词至少 2 条不同短语搭配，否则标记未完成

## 十一、动词人称覆盖范围：从 4 人称扩展为全 6 人称

原文档第10节"第一阶段重点人称"只覆盖 yo / tú / él-ella-usted / nosotros，现调整为**全部 6 个人称**：

- yo（我）
- tú（你）
- él / ella / usted（他/她/您）
- nosotros / nosotras（我们）
- vosotros / vosotras（你们，西班牙本土用法）
- ellos / ellas / ustedes（他们/她们/您们）

**原因**：只会前 4 个人称能表达自己、对话对方，但无法转述、描述第三方、讲故事——这类场景在真实生活里同样高频，不能算"够用"。

**落地方式**：

- 每个必背动词（原文档第10节列出的 ser / estar / tener / querer / poder 等）需要补齐全部 6 个人称块
- 每个人称块仍然是独立 entry，遵循已有 JSON schema，不改变数据结构
- **不要求一次性全部补齐**：先按第10节原有优先级（yo/tú/él/nosotros）建完 Core 100，再在扩展到 Core 300 时把 vosotros / ellos 补上，避免初期库爆炸式增长
- 每日 20 个的训练量不变，人称块数量增加只会拉长间隔复习覆盖全部内容所需的总时长，不改变每日节奏

---

## 十二、场景定位：不做功能，只做内容筛选坐标系

场景**不作为独立训练模式或页面**，这一点维持原设计（原文档第19节已明确"场景句不是独立训练模式，只是基础库里的短句"）。

但场景清单本身需要修正：原文档第19节列的酒吧/超市/问路/药店是**游客场景**，作为在西班牙实际生活的人，真正高频、且背不出来会真的卡住的是**居民场景**：

- 银行（开户、转账、卡的问题）
- 房东/租房（报修、续约、押金、提前通知）
- 官方手续（empadronamiento、NIE、社保、预约窗口）
- 医疗（预约、描述症状、药物过敏）
- 快递/包裹（取件、地址问题、退货）
- 电话/网络套餐（客服沟通、投诉、更改套餐）

这些场景语言难度更高（更长的解释性表达、被动语态、官方用语），词块类型也和游客场景不同。**场景清单只用于指导 Core 300/500 阶段该往哪些方向补内容，不涉及 UI/页面结构改动。**

---

## 十三、自定义词库添加：App 外生成 + 人工审核 + 批量导入

**背景**：居民场景的内容永远无法靠预设词库穷尽，需要一个补充入口，让日常生活中冒出来的具体需求（比如房东突然要涨租）能被及时收录。

**为什么不做 App 内"AI 一键生成并添加"功能**：

1. **违反"宁少勿错"原则**——错误数据会被间隔复习系统反复强化，AI生成内容如果不经审核直接进入训练队列，一旦有错（性别、搭配不自然、翻译腔），比不学还糟。
2. **是一个 ADHD 陷阱**——"添加新内容"天然比"完成训练"更有新鲜感，一旦有便捷入口，容易演变成不断生成/添加而不是老实完成每日20个，架空"每日新内容上限5个"这道防线。

**实际流程（在 App 外完成生成和初审）**：

1. 在 ChatGPT/Claude 等工具里描述场景，让其按 JSON schema 生成 entry（category、zh、es、accepted_answers、example_es、note 等字段完整）
2. 自己过一遍确认无明显错误、符合"不孤立"规则（动词要有人称块+搭配，名词要有搭配句）
3. 通过基础库页面一个低调的"批量导入"入口粘贴 JSON，系统校验格式后预览，确认无误再导入

**UI 原则**：导入入口必须弱化处理——放在基础库页面底部，视觉上明显低于"开始背诵"等日常操作，不能设计成吸引人的主要 CTA，避免它和每日训练节奏抢注意力。

## 项目定位说明

这是一个**针对个人使用特性设计的自用工具**，不是商业产品。所有设计取舍都以"是否真的会被我自己长期使用"为唯一标准，不考虑用户增长、留存率、通用性等商业产品逻辑。这一点会持续影响后续所有功能判断——凡是增加操作摩擦、制造焦虑源、或者提供"看起来有用但实际会被用来逃避训练"的功能，即使在别的产品里合理，在这里也应该被砍掉。

## 十五、静默计时（用于自我记录，不用于评判或激励）

**目的**：给自己一个客观、可信的"学了多久"的记录，用于自我满足感，不是效率考核。

**设计**：

- **自动启动**：进入训练页第一题时静默开始计时，不需要手动点"开始"
- **训练过程中不显示任何计时界面**——不显示跳动的秒表、不显示进度，纯后台记录，界面上只有一个"结束"按钮
- **结束按钮**：点击后停止并记录本次时长，按钮不常驻视觉焦点，是一个小号次要按钮
- **忘记结束的兜底逻辑**：若用户未点击结束就离开或关闭 App，系统以**最后一次答题的时间戳**作为实际结束时间，而不是以离开/关闭的时间为准。这是为了保证记录的诚实性——避免因为忘记结束导致时长虚高，产生一个不可信的数字，反而重新制造自我怀疑
- **统计维度**：只做**日 / 周 / 月 / 年**四档聚合，在统计页用标签切换查看，不设目标、不与历史对比、不因时长短触发任何提示文案。纯记录，不做评判

## 未改变的部分

原文档的以下设计保持不变，本次讨论确认方向正确：

- 复制给 GPT 生成复现小课文的机制
